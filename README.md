# Chess Game Collection / 棋类游戏合集

一个基于 **C++17、Visual Studio 2022 和 EasyX 图形库** 开发的桌面棋类游戏合集。项目通过统一欢迎大厅整合了 **中国象棋、国际象棋、围棋、五子棋、飞行棋和跳棋** 六种游戏，并为不同棋类提供计时、真人对战、机器人对战、局势分析、操作提示、悔棋或撤销、日志记录等功能。

当前版本为 **v1.3**。项目并不是把六个互不相关的程序简单放入同一个文件夹，而是采用“统一入口 + 公共基础模块 + 各棋类独立规则模块”的结构进行整合。其中，**国际象棋、围棋和五子棋三类棋的前置基础代码已经统一**：三者共同使用 `qilei_common.h/.cpp` 中的设置界面、对局计时、日志记录、机器人模式、难度换算、工程路径定位和外部引擎进程通信逻辑；各自只保留棋盘规则、协议转换、局势分析和专项界面代码。

## 2. 项目简介

程序启动后首先进入“棋类游戏大厅”。大厅以六张卡片展示当前支持的游戏：

1. 中国象棋；
2. 国际象棋；
3. 围棋；
4. 五子棋；
5. 飞行棋；
6. 跳棋。

玩家可以使用鼠标点击卡片，也可以使用方向键切换选项并按 `Enter` 进入。每种游戏结束或退出后，程序会返回统一大厅，而不是直接关闭整个程序。

不同棋类使用不同的规则与机器人实现：

- 中国象棋使用本项目独立规则模块，并支持 Pikafish 高质量机器人；
- 国际象棋实现标准规则，并优先调用 Stockfish 18；
- 围棋实现 19 路棋盘、中国数子规则与 7.5 目贴目，并优先调用 KataGo；
- 五子棋实现 15 路无禁手规则，并优先调用 Rapfi；
- 飞行棋使用本地启发式机器人、计时、特殊格和完整排名；
- 跳棋使用标准 121 孔星形棋盘、BFS 连续跳跃、机器人决策与最终区域保护。

项目中的外部引擎均作为独立进程运行，游戏规则和界面仍由本项目自己的 C++ 代码负责。外部引擎无法启动时，国际象棋、围棋和五子棋可以使用各自的内置后备机器人继续运行；中国象棋机器人则需要对应的 Pikafish 引擎文件。

## 3. 特别说明：三类棋共用统一前置代码

> **国际象棋、围棋和五子棋前面那一大部分基础代码是统一的，不是三套完全重复的实现。**

这三类棋共同包含以下使用流程：

```text
进入统一设置页
      ↓
选择每步时长
      ↓
选择单方总时长
      ↓
选择双方玩家 / 单方机器人 / 双方机器人
      ↓
选择机器人难度
      ↓
创建统一计时器与统一日志
      ↓
按需要启动外部引擎进程
      ↓
进入各自棋盘与规则模块
```

它们共同依赖：

```text
daima/qilei_common.h
daima/qilei_common.cpp
```

### 3.1 统一设置结构

三类棋共同使用：

```cpp
struct qilei_game_setting
{
    int qilei_step_seconds;
    int qilei_total_seconds;
    int qilei_robot_mode;
    int qilei_robot_level;
};
```

统一保存：

- 每步允许时间；
- 每方总时间；
- 人机对战模式；
- 机器人难度。

共同可选设置为：

| 设置项 | 可选值 |
| --- | --- |
| 每步时长 | 30 秒、60 秒、90 秒 |
| 单方总时长 | 5 分钟、10 分钟、15 分钟、20 分钟 |
| 对战模式 | 双方玩家、先手机器人、后手机器人、双方机器人 |
| 机器人难度 | 入门、进阶、大师 |

### 3.2 统一计时器

三类棋共同使用 `qilei_clock`：

- 同时记录当前步骤剩余时间；
- 分别记录双方总剩余时间；
- 根据真实经过毫秒数扣时；
- 每次成功走棋后切换计时方；
- 步时或总时耗尽时返回超时方；
- 游戏结束时停止计时；
- 将时间格式化为 `mm:ss`。

### 3.3 统一日志系统

三类棋共同使用 `qilei_result_logger`：

- 自动定位项目根目录；
- 在统一的 `result/` 目录中创建日志；
- 日志名采用“棋类名称 + 时间”；
- 使用 UTF-8 with BOM 保存中文；
- 记录设置、机器人名称、走棋、分析、悔棋、投降、超时和结果。

### 3.4 统一外部引擎进程

三类棋共同使用 `qilei_engine_process`：

- 使用 Windows 匿名管道连接外部引擎；
- 使用 `CreateProcessW()` 创建无窗口子进程；
- 向引擎标准输入发送命令；
- 从标准输出读取返回内容；
- 支持等待指定协议标记；
- 支持超时检测；
- 退出时发送结束命令并释放进程和管道句柄。

虽然国际象棋、围棋和五子棋使用的协议不同，但底层进程管理已经统一：

| 棋类 | 外部引擎 | 通信协议 |
| --- | --- | --- |
| 国际象棋 | Stockfish 18 | UCI |
| 围棋 | KataGo 1.16.5 | GTP |
| 五子棋 | Rapfi 250615 | Piskvork |

### 3.5 统一难度换算

三类棋共同通过：

```cpp
qilei_robot_think_ms()
```

把“入门、进阶、大师”转换为机器人思考时间，并同时受到当前每步时长限制。

### 3.6 三类棋真正不同的部分

统一前置代码不代表三种棋的规则代码完全相同。它们各自仍然保留：

- 独立棋盘数据结构；
- 独立合法走法判断；
- 独立胜负规则；
- 独立外部协议坐标转换；
- 独立局势分析；
- 独立界面绘制；
- 独立后备机器人。

因此，本项目结构可以概括为：

```text
统一设置、计时、日志、引擎进程
                +
国际象棋 / 围棋 / 五子棋各自规则和协议
```

这种写法可以减少重复代码，也便于后续统一修改计时、机器人难度、日志格式和引擎启动方式。

## 4. 统一棋类大厅

### 4.1 大厅功能

`qilei_menu` 负责六种棋类的统一欢迎页：

- 1180 × 760 图形窗口；
- 六张棋类卡片；
- 每种棋类独立主题颜色；
- 显示棋类名称和主要功能；
- 支持鼠标悬停；
- 支持鼠标点击进入；
- 支持方向键切换；
- 支持 `Enter` 进入；
- 支持 `Esc` 退出程序。

### 4.2 统一程序入口

`daima/main.cpp` 只负责：

1. 创建统一菜单对象；
2. 获取玩家选择的棋类编号；
3. 创建对应游戏对象；
4. 调用对应 `run()` 接口；
5. 游戏退出后再次显示大厅。

统一入口没有直接实现任何棋类规则，避免主函数与具体游戏逻辑耦合。

### 4.3 游戏调度关系

```text
main()
  └─ qilei_menu::qilei_choose()
      ├─ zhongguoxiangqi_game
      ├─ guojixiangqi_game
      ├─ weiqi_game
      ├─ wuziqi_game
      ├─ feixingqi_game
      └─ tiaoqi_game
```

## 5. 中国象棋

### 5.1 基础规则

中国象棋模块实现：

- 标准 10 × 9 棋盘；
- 红黑双方 32 枚棋子；
- 帅/将九宫限制；
- 仕/士斜走；
- 相/象田字、不能过河和塞象眼；
- 马走日字和蹩马腿；
- 车直线移动；
- 炮隔子吃子；
- 兵/卒过河规则；
- 将帅照面；
- 将军判断；
- 不能送将；
- 吃帅结束；
- 无合法走法判负；
- 悔棋；
- 每步与单方总计时；
- 超时判负；
- 本地局势分析。

### 5.2 人机模式

中国象棋支持：

- 红黑双方玩家；
- 红方机器人；
- 黑方机器人；
- 红黑双方机器人。

机器人模块使用 Pikafish 高质量象棋引擎，并通过后台搜索避免阻塞 EasyX 主循环。引擎走法转换为内部坐标后，还会再次调用本项目合法走法函数校验。

### 5.3 当前压缩包的重要说明

源码会从以下目录查找中国象棋引擎：

```text
engines/zhongguoxiangqi/
```

但本次上传的压缩包中没有实际包含该目录及 Pikafish 程序和 NNUE 文件。

因此：

- 中国象棋双人模式仍可使用；
- 中国象棋规则、计时、局势分析和悔棋不受影响；
- 选择中国象棋机器人模式时，需要先补齐 Pikafish 引擎目录；
- 未补齐时，程序会显示引擎启动错误。

建议目录至少包含：

```text
engines/zhongguoxiangqi/
├── pikafish.exe
├── pikafish.nnue
├── Copying.txt
├── NNUE-License.md
├── SOURCE.txt
└── AUTHORS
```

## 10. 跳棋

### 10.1 棋盘与走法

跳棋模块实现：

- 标准 121 孔星形棋盘；
- 2–6 名玩家；
- 每人 3、6 或 10 个棋子；
- 相邻一步移动；
- 连续跳跃；
- BFS 搜索所有跳跃落点；
- 自己最终区只进不出；
- 限制进入其他玩家最终区；
- 外来棋子只能向出口撤离；
- 落点二次确认；
- 完整排名；
- 超时判负。

### 10.2 机器人

跳棋支持：

- 0 至全部玩家机器人；
- 五秒分阶段自动走棋；
- 推进评分；
- 起始区撤离；
- 自己最终区整理；
- 对方最终区撤离；
- 重复局面检测；
- 本方布局重复检测；
- 重复边惩罚；
- 立即反向走法惩罚；
- 高质量候选加权随机；
- 连续无推进时强制优先前进。

### 10.3 最终区域保护

- 最终区域达到 60% 完成数量后进入保护；
- 如果区域内仍有其他玩家棋子，滞留玩家判负；
- 机器人在区域主人达到约 30% 时提前规划撤离；
- 判负玩家棋子从棋盘清除；
- 记录判负原因、完成数量和最终名次。

### 10.4 局势分析

跳棋局势分析考虑：

- 棋子推进；
- 跨越中线；
- 己方跳跃配合；
- 进入最终区；
- 是否给对手提供跳板；
- 完成和淘汰状态。

## 14. 主要架构与实现思路

### 14.1 三层结构

项目可以分成三层：

```text
第一层：统一大厅
第二层：公共基础能力
第三层：各棋类规则与界面
```

### 14.2 入口适配器

每种棋类都提供独立 `entry`：

```cpp
class xxx_game
{
public:
    int xxx_run();
};
```

统一大厅不需要了解棋盘内部数据，只调用统一的 `run()` 接口。

### 14.3 外部引擎适配

三种外部协议都建立在统一进程层之上：

```text
qilei_engine_process
      ├─ UCI：Stockfish
      ├─ GTP：KataGo
      └─ Piskvork：Rapfi
```

游戏专项模块负责：

- 发送协议初始化命令；
- 将棋盘历史转换成协议坐标；
- 解析引擎返回；
- 转换为内部落点；
- 校验落点；
- 执行走法。

### 14.4 日志目录统一

所有游戏都将日志保存到：

```text
result/
```

日志文件名包含棋类名称和时间。这样可以避免六种游戏分别在不同目录生成难以管理的日志。

### 14.5 EasyX 窗口切换

每次进入游戏前关闭大厅窗口，游戏结束后关闭当前棋类窗口，再由主循环重新创建大厅窗口。这种方式避免不同窗口尺寸和批量绘图状态互相影响。

## 16. 已知问题与后续优化

### 16.1 中国象棋引擎文件缺失

当前压缩包未包含：

```text
engines/zhongguoxiangqi/
```

这是当前最明确的发布缺口。后续发布时应补齐 Pikafish、NNUE 和许可证文件，或为中国象棋增加本地后备机器人。

### 16.2 公共框架覆盖范围有限

目前统一 `qilei_common` 主要服务国际象棋、围棋和五子棋。中国象棋、飞行棋和跳棋仍保留原有独立计时、日志和设置模块。

后续可以继续抽象：

- 通用计时接口；
- 通用日志接口；
- 通用机器人状态；
- 通用返回大厅按钮；
- 通用提示和悔棋按钮；
- 通用局势条组件。

### 16.3 外部引擎体积较大

围棋模型、Stockfish 和 Rapfi 权重会显著增加项目体积。公开仓库时应根据实际发布方式处理大型二进制和模型文件，并完整保留许可证。

### 16.4 引擎思考可能阻塞

国际象棋、围棋和五子棋公共引擎接口当前主要采用同步等待。后续可以参考中国象棋的后台异步搜索方式，减少复杂局面下的界面等待。

### 16.5 规则完整性

- 中国象棋尚未覆盖完整竞赛和棋规则；
- 国际象棋可继续完善三次重复和不足子力和棋；
- 围棋当前使用简单劫，可扩展超级劫；
- 五子棋当前为无禁手规则，可增加连珠规则；
- 飞行棋可增加击退；
- 跳棋可增加规则配置开关。

### 16.6 保存与回放

当前主要提供文本日志，没有统一的棋谱保存、读档和自动回放功能。

### 16.7 界面统一

六种游戏的来源和发展阶段不同，界面风格仍有差异。后续可统一：

- 标题栏；
- 右侧信息面板；
- 按钮颜色；
- 字体；
- 返回大厅；
